# Lecture 2 — PKCE, JWT Anatomy, and the Auth-Code Flow Walked

> *In which we open the hood. The PKCE math is one SHA-256 and one base64url encode. The JWT format is three base64url segments separated by dots, of which only the signature actually protects you. We walk the authorization-code-with-PKCE flow byte by byte against a running Keycloak instance, decode an ID token by hand in the browser console, verify its signature with the `jose` library, and demonstrate what happens to a tampered token. By the end of this lecture you will be able to read a JWT, find its `kid`, fetch the matching key from the JWKS endpoint, and verify the signature yourself.*

---

## 1. PKCE — Proof Key for Code Exchange (RFC 7636)

PKCE (pronounced "pixie") is the extension that makes the authorization-code grant safe for public clients. The full spec is RFC 7636, published in September 2015. It is short — about 22 pages, of which roughly six are normative — and you should read it once. The mechanism is short enough that you could implement it from scratch in 50 lines.

### 1.1 The threat PKCE defends against

Before PKCE, the authorization-code flow had a hole when used by mobile apps and SPAs. The hole was the redirect leg.

The scenario: you have installed a legitimate banking app and a malicious app on the same phone. Both apps register the custom URI scheme `bankapp://callback` (Android and iOS used to allow this — modern OS versions clamped down, but the threat was real). The legitimate app starts the OAuth flow, redirects the user to the bank's authorization server. The user authenticates. The bank redirects to `bankapp://callback?code=ABC123`. The OS routes the redirect to **whichever app responds first**, and the malicious app responds first. The malicious app now has the authorization code.

In a confidential-client flow, the malicious app cannot exchange the code at the token endpoint without the client secret — and the client secret is held only by the legitimate app's backend server. The exchange fails. Crisis averted.

In a public-client flow, **there is no client secret**, because a public client cannot keep one. The malicious app POSTs the code to the token endpoint with the legitimate client's ID and successfully exchanges it for tokens. The malicious app is now logged in as the user.

The SPA equivalent: any code-level vulnerability that exposes the URL of the callback page (a permissive CORS policy, a redirect injection on another page on the same origin, a logged request appearing in an error monitor) leaks the code to an attacker who can complete the exchange.

PKCE closes the hole by binding the token exchange to a secret that **the legitimate client generated and only the legitimate client knows**.

### 1.2 The math

PKCE adds two parameters to the authorization request and one to the token request. The math:

1. **Generate a code_verifier.** A high-entropy random string, 43 to 128 characters from the unreserved character set `[A-Za-z0-9-._~]`. The spec recommends 32 bytes of random data, base64url-encoded (which yields 43 characters).

```javascript
function generateVerifier() {
  const bytes = new Uint8Array(32);
  crypto.getRandomValues(bytes);
  return base64UrlEncode(bytes);  // 43 chars
}
```

2. **Derive the code_challenge.** SHA-256 hash the verifier, then base64url-encode the hash.

```javascript
async function deriveChallenge(verifier) {
  const data = new TextEncoder().encode(verifier);
  const hash = await crypto.subtle.digest('SHA-256', data);
  return base64UrlEncode(new Uint8Array(hash));
}
```

3. **Send the challenge in the authorization request.**

```
GET /authorize
  ?response_type=code
  &client_id=spa-client
  &redirect_uri=https://app/callback
  &scope=openid profile
  &state=<random>
  &code_challenge=<challenge>
  &code_challenge_method=S256
```

4. **Send the verifier in the token request.**

```
POST /token
  grant_type=authorization_code
  code=<code-from-callback>
  redirect_uri=https://app/callback
  client_id=spa-client
  code_verifier=<the-original-verifier>
```

5. **The authorization server verifies.** It re-derives `BASE64URL(SHA256(received_verifier))` and compares to the `code_challenge` it received in step 3. If they match, it issues tokens. If they do not, it returns an error.

### 1.3 Why this works

The authorization code by itself is no longer enough to exchange for tokens — you also need the verifier. The verifier never appears in the redirect. The verifier was generated by the legitimate client and stored only in the legitimate client's `sessionStorage`. A malicious app that intercepts the code has no way to know the verifier; the SHA-256 hash is one-way; the token exchange fails.

The security property is provable under the standard assumptions about SHA-256: an attacker who knows the challenge but not the verifier cannot produce a valid verifier without breaking SHA-256.

### 1.4 The `plain` method (do not use)

RFC 7636 §4.2 also defines a `plain` method where `code_challenge = code_verifier` — no hashing. This is provided for clients that cannot do SHA-256, which is approximately none in 2026. **Always use `S256`.** Keycloak, Auth0, and every other modern IdP support `S256` and will accept `plain` only if you explicitly configure it to.

### 1.5 PKCE in 30 lines

Here is the full implementation, suitable for putting in a utility file:

```javascript
function base64UrlEncode(arrayBuffer) {
  let str = '';
  const bytes = new Uint8Array(arrayBuffer);
  for (let i = 0; i < bytes.byteLength; i++) {
    str += String.fromCharCode(bytes[i]);
  }
  return btoa(str)
    .replace(/\+/g, '-')
    .replace(/\//g, '_')
    .replace(/=+$/, '');
}

function generateVerifier() {
  const bytes = new Uint8Array(32);
  crypto.getRandomValues(bytes);
  return base64UrlEncode(bytes);
}

async function deriveChallenge(verifier) {
  const data = new TextEncoder().encode(verifier);
  const hash = await crypto.subtle.digest('SHA-256', data);
  return base64UrlEncode(hash);
}

export async function startPkceFlow() {
  const verifier = generateVerifier();
  const challenge = await deriveChallenge(verifier);
  sessionStorage.setItem('pkce_verifier', verifier);
  return challenge;
}
```

Exercise 2 walks you through this from scratch. The library handles it in production; you should be able to read what the library does.

## 2. JWT — JSON Web Token (RFC 7519)

A JSON Web Token is a compact, URL-safe representation of claims about a subject, signed by the issuer. RFC 7519 is short (30 pages) and self-contained. The format is built on top of JOSE (RFC 7515 for signatures, RFC 7517 for keys, RFC 7518 for algorithms).

### 2.1 The format

A JWT is three base64url-encoded segments separated by dots:

```
<header>.<payload>.<signature>
```

Each segment is a base64url-encoded JSON object (for the header and payload) or a base64url-encoded binary signature.

A concrete example, taken from a Keycloak token (truncated for readability):

```
eyJhbGciOiJSUzI1NiIsInR5cCIgOiAiSldUIiwia2lkIiA6ICJrZXktMDAxIn0
.
eyJleHAiOjE3MzUwMDAwMDAsImlhdCI6MTczNDk5OTQwMCwianRpIjoiYWJjMTIzIiwiaXNzIjoiaHR0cDovL2xvY2FsaG9zdDo4MDgwL3JlYWxtcy9jcnVuY2giLCJhdWQiOiJzcGEtY2xpZW50Iiwic3ViIjoiYWxpY2UiLCJ0eXAiOiJJRCIsImF6cCI6InNwYS1jbGllbnQiLCJub25jZSI6InhqaG56WCJ9
.
PnLkLDpLBn5n4wQg... (binary signature, base64url encoded)
```

Decode the header in the browser console:

```javascript
const token = "eyJhbGciOiJSUzI1NiIsInR5cCIgOiAiSldUIiwia2lkIiA6ICJrZXktMDAxIn0...";
const [headerB64] = token.split('.');
const header = JSON.parse(atob(headerB64.replace(/-/g, '+').replace(/_/g, '/')));
console.log(header);
// { alg: "RS256", typ: "JWT", kid: "key-001" }
```

Decode the payload the same way:

```javascript
const [, payloadB64] = token.split('.');
const payload = JSON.parse(atob(payloadB64.replace(/-/g, '+').replace(/_/g, '/')));
console.log(payload);
// {
//   exp: 1735000000,
//   iat: 1734999400,
//   jti: "abc123",
//   iss: "http://localhost:8080/realms/crunch",
//   aud: "spa-client",
//   sub: "alice",
//   typ: "ID",
//   azp: "spa-client",
//   nonce: "xjhnzX"
// }
```

The signature is binary; reading it by eye is useless. The library re-derives it from the header + payload using the issuer's public key and compares.

**The critical observation: anyone can read the payload.** The protection is not confidentiality. The protection is the signature, which guarantees that the payload has not been altered since the issuer signed it. We will see in §3 how the signature is verified.

### 2.2 The header

The header tells the verifier how to verify. The mandatory and common fields:

- **`alg`** — the signing algorithm. Values from RFC 7518 §3 — `HS256` (HMAC + SHA-256, symmetric), `RS256` (RSA + SHA-256, asymmetric), `ES256` (ECDSA on P-256 + SHA-256, asymmetric), `PS256` (RSA-PSS + SHA-256, asymmetric), `none` (no signature — must be rejected).
- **`typ`** — the token type. Typically `JWT`. Some specs require `at+jwt` for access tokens (RFC 9068).
- **`kid`** — the key ID. The issuer publishes a JWKS (set of keys) at `${iss}/.well-known/jwks.json` or wherever discovery points; the `kid` tells you which key in that set was used to sign. The issuer can rotate keys without breaking outstanding tokens.

### 2.3 The payload — registered claims (RFC 7519 §4.1)

The IETF reserved a small set of claim names with defined semantics:

- **`iss`** (issuer) — the URL of the issuer. For Keycloak: `http://localhost:8080/realms/<realm>`. **Always verify** this matches the issuer you expect.
- **`sub`** (subject) — the user's stable ID at the issuer. The single most important claim. Use it as the foreign key in your database. **Do not use `email` for this**: emails change.
- **`aud`** (audience) — the intended recipient. For ID tokens, this is your `client_id`. For access tokens, this is typically the resource server's identifier. **Always verify**.
- **`exp`** (expiration) — Unix timestamp after which the token must be rejected.
- **`nbf`** (not-before) — Unix timestamp before which the token must be rejected (rarely used, but check it if present).
- **`iat`** (issued-at) — Unix timestamp when the token was issued.
- **`jti`** (JWT ID) — a unique identifier for the token. Useful for denylisting.

### 2.4 The payload — OIDC claims

OIDC Core §2 adds:

- **`auth_time`** — Unix timestamp when the user authenticated. Useful when your app cares about "how recently did this user prove they are who they say they are" (e.g., re-auth before a sensitive action).
- **`nonce`** — the nonce the client sent in the authorization request. Must match.
- **`acr`** (Authentication Context Class Reference) — the level of assurance of the authentication. E.g., `urn:mace:incommon:iap:silver` for a low-assurance flow.
- **`amr`** (Authentication Methods References) — the methods used. E.g., `["pwd"]` for password, `["pwd", "otp"]` for password plus TOTP, `["pwd", "mfa"]` for password plus any second factor.
- **`azp`** (Authorized Party) — when `aud` contains multiple values, `azp` names the specific party the token was issued to.
- **`at_hash`** — a hash of the access token, useful for binding the ID token to the access token.

### 2.5 The payload — profile / email claims (per scope)

When the client requests the `profile` or `email` scope, the IdP includes (depending on what is on file for the user):

- `name`, `given_name`, `family_name`, `middle_name`, `nickname`, `preferred_username`
- `profile`, `picture`, `website`
- `gender`, `birthdate`, `zoneinfo`, `locale`
- `email`, `email_verified`
- `phone_number`, `phone_number_verified`
- `address` (a structured object)
- `updated_at`

These are convenience claims for the client to display the user's name, avatar, etc. **Treat the values as cosmetic; do not use them for authorization decisions.** Use `sub`.

### 2.6 What NOT to put in a claim

A short list. Memorize it.

1. **Passwords.** Ever. Under no circumstances. JWTs are signed but not encrypted; the payload is readable by anyone who has the token.
2. **API keys.** Same reason.
3. **Session secrets.** Same reason.
4. **Personally identifiable information you would not want logged.** Tokens end up in log files, error monitors, and analytics traces. If you would refuse to log a Social Security Number, do not put it in a JWT.
5. **Anything large.** JWTs travel in `Authorization` headers and in cookies; headers and cookies have size limits (8 KB is a common limit). Keep tokens small.
6. **Anything that changes often.** Each change requires the user to re-authenticate to get a new token. Roles that change weekly are fine; user preferences that change every page view are not.
7. **Authorization decisions you have not made yet.** If your access policy is "user can read documents they own," do not encode the list of document IDs in the token. Encode the user ID (`sub`) and look up ownership at request time.

The general rule: a JWT is a **verified identity assertion**. Keep its payload to the assertion itself — who the user is, what scopes they were granted, when the token expires.

## 3. Signature verification

The signature is the only thing that makes a JWT trustworthy. We walk verification in detail because most security failures in JWT-using systems come from skipped or sloppy verification.

### 3.1 The high-level algorithm

```
1. Split the token: header.payload.signature.
2. Decode the header.
3. Read header.kid. Read header.alg.
4. Fetch the JWKS from the issuer (cached).
5. Find the key in the JWKS whose kid matches header.kid.
6. Verify header.alg is one of the algorithms YOU permit
   (do NOT trust header.alg alone — see 3.4).
7. Re-derive the signature over (header.payload) using the key.
8. Compare the re-derived signature to the one in the token.
9. If equal, the token has not been tampered with. Proceed to claim validation.
10. Validate iss, aud, exp, nbf, iat, nonce against expected values.
```

### 3.2 The JWKS endpoint

The JSON Web Key Set is a list of keys the issuer is currently using to sign tokens. For Keycloak, the URL is `${iss}/protocol/openid-connect/certs`; the OIDC discovery document's `jwks_uri` field points to it. A trimmed example:

```json
{
  "keys": [
    {
      "kid": "key-001",
      "kty": "RSA",
      "alg": "RS256",
      "use": "sig",
      "n": "x7VbR-XL... (the RSA public modulus, base64url-encoded)",
      "e": "AQAB"
    },
    {
      "kid": "key-002",
      "kty": "RSA",
      "alg": "RS256",
      "use": "sig",
      "n": "...",
      "e": "AQAB"
    }
  ]
}
```

**The JWKS contains only public keys.** It is safe to publish; the corresponding private keys live on the issuer's server and never leave.

The presence of two keys is intentional. During key rotation, the issuer:

1. Publishes the new key in JWKS (now there are two).
2. Starts signing new tokens with the new key.
3. Waits for all outstanding tokens signed with the old key to expire.
4. Removes the old key from JWKS.

Clients that cache the JWKS need to re-fetch on a `kid` miss; the cache strategy is the subject of Challenge 2.

### 3.3 Using the `jose` library

Verification by hand is instructive once. In production you use a library. The `jose` package (<https://github.com/panva/jose>) is the de-facto standard:

```typescript
import { jwtVerify, createRemoteJWKSet } from 'jose';

const JWKS = createRemoteJWKSet(
  new URL('http://localhost:8080/realms/crunch/protocol/openid-connect/certs')
);

const { payload, protectedHeader } = await jwtVerify(token, JWKS, {
  issuer: 'http://localhost:8080/realms/crunch',
  audience: 'spa-client',
  algorithms: ['RS256'],   // do NOT trust the alg in the token alone
  clockTolerance: 5,        // 5 seconds of skew
});

console.log(payload.sub, payload.email);
```

`createRemoteJWKSet` handles JWKS fetching, caching, and the kid-miss re-fetch. The `algorithms` array is the algorithm allowlist; **always pin it**.

### 3.4 The historic JWT vulnerabilities

The early JWT-library ecosystem had three notorious vulnerabilities that you should know by name, both to recognize them in old code and to understand why your library is so paranoid.

**`alg: none`.** RFC 7519 §6.1 allows `alg: none` for unsigned tokens. Some libraries' early default was "if `alg: none`, skip signature verification and accept the token as valid." An attacker would forge a token with `alg: none`, a payload of their choosing, and an empty signature segment; the library would accept it. The fix is to **never accept `alg: none` for verification** — your allowlist must not include it. The `jose` library has never permitted `alg: none` in the verify path.

**HS256/RS256 confusion.** Some libraries asked "give me the verification key" and used the same code path for HMAC (symmetric) and RSA (asymmetric). The verifier's RSA public key — which is public, by design — would be used as the HMAC key. An attacker who had the public key (everyone has the public key) could sign tokens with HS256 using the public key as the HMAC secret, and the verifier would accept them. **The fix is to pin the algorithm**: the verifier specifies `algorithms: ['RS256']`, and the library refuses to use the RSA key for HMAC.

**Weak HS256 keys.** When the HS256 signing key is a short string ("secret123"), an offline brute-force attack can recover it from a captured token. The recovered key lets the attacker sign their own tokens. The fix is to **either use a high-entropy HMAC key** (32+ random bytes) **or switch to an asymmetric algorithm** (RS256, ES256, PS256). For tokens that cross a trust boundary — between your IdP and your relying parties — asymmetric is the correct choice.

You will encounter older code that does not protect against these. Recognize them. Fix them.

### 3.5 The minimal correct verifier

For reference, a verifier that defends against the three historic attacks:

```typescript
import { jwtVerify, createRemoteJWKSet } from 'jose';

const ISSUER = 'http://localhost:8080/realms/crunch';
const AUDIENCE = 'spa-client';
const ALGORITHMS = ['RS256'];

const JWKS = createRemoteJWKSet(new URL(`${ISSUER}/protocol/openid-connect/certs`));

export async function verifyIdToken(token: string, expectedNonce: string) {
  const { payload } = await jwtVerify(token, JWKS, {
    issuer: ISSUER,
    audience: AUDIENCE,
    algorithms: ALGORITHMS,
    clockTolerance: 5,
  });

  if (payload.nonce !== expectedNonce) {
    throw new Error('Nonce mismatch — possible replay attack');
  }

  return payload;
}
```

That is roughly the verification code in the mini-project. Exercise 1 walks you through writing this from scratch, deliberately, including the manual base64url decoding step.

## 4. The full PKCE flow against Keycloak

We now walk the entire flow against a running Keycloak instance, with the actual HTTP requests and responses you will see in DevTools. The mini-project's `docker-compose.yml` brings up Keycloak on `localhost:8080`; the mini-project's seed realm is `crunch` and seed user is `alice` / `alice-password`.

### 4.1 Step 1 — The authorize redirect

The SPA navigates the browser to:

```
http://localhost:8080/realms/crunch/protocol/openid-connect/auth
  ?response_type=code
  &client_id=spa-client
  &redirect_uri=http%3A%2F%2Flocalhost%3A5173%2Fcallback
  &scope=openid%20profile%20email
  &state=qzkBFRzCXt
  &nonce=xJ2nHpqr
  &code_challenge=hxR-i9NM2EmH8tEbgczO3-AlV67otsAVOykMC-K7lr0
  &code_challenge_method=S256
```

(Line-broken for readability; in reality this is one URL.) The client has, in `sessionStorage`:

- `pkce_verifier` — the random verifier whose SHA-256 was sent as the challenge.
- `oidc_state` — the `state` value.
- `oidc_nonce` — the `nonce` value.

### 4.2 Step 2 — User authenticates at Keycloak

Keycloak shows its login page. The user enters `alice` / `alice-password` and submits. Keycloak validates the password.

(If MFA is enabled — see lecture 3 — Keycloak then prompts for the TOTP code.)

### 4.3 Step 3 — Keycloak redirects back

```
HTTP/1.1 302 Found
Location: http://localhost:5173/callback?code=ABC123.DEF456&state=qzkBFRzCXt
```

The client's callback route reads `code` and `state` from the URL. **It verifies `state` against `sessionStorage.oidc_state`.** If they do not match, the callback fails.

### 4.4 Step 4 — Token exchange

The SPA POSTs to the token endpoint:

```
POST http://localhost:8080/realms/crunch/protocol/openid-connect/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=ABC123.DEF456
&redirect_uri=http%3A%2F%2Flocalhost%3A5173%2Fcallback
&client_id=spa-client
&code_verifier=<the-verifier-from-sessionStorage>
```

Note the absence of a `client_secret`. The SPA is a public client; the verifier replaces the secret.

### 4.5 Step 5 — Token response

```
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: no-store
Pragma: no-cache

{
  "access_token": "eyJhbGciOiJSUzI1NiIs...",
  "expires_in": 900,
  "refresh_expires_in": 1800,
  "refresh_token": "eyJhbGciOiJIUzUxMiI...",
  "token_type": "Bearer",
  "id_token": "eyJhbGciOiJSUzI1NiIs...",
  "not-before-policy": 0,
  "session_state": "5e8c8a3b-1234-...",
  "scope": "openid email profile"
}
```

Three tokens come back:

- **`access_token`** — a JWT (Keycloak default; configurable) that you send to the resource server. Lifetime 900 s = 15 min.
- **`id_token`** — a JWT describing the authenticated user. You verify this and use its claims for display and for `sub`.
- **`refresh_token`** — used to obtain new access and id tokens when this access token expires. Lifetime 30 min in the seed realm; production typically uses hours to days.

### 4.6 Step 6 — Calling the resource server

```
GET http://localhost:3001/api/me
Authorization: Bearer eyJhbGciOiJSUzI1NiIs...
```

The resource server verifies the access token's signature against the same JWKS, checks `iss` and `aud`, and returns the user's data. (The mini-project's resource server is a 50-line Express app under `mini-project/starter/resource-server/` — see the README.)

### 4.7 Step 7 — Silent renew

When the access token nears expiry, the SPA renews silently:

```
POST http://localhost:8080/realms/crunch/protocol/openid-connect/token

grant_type=refresh_token
&client_id=spa-client
&refresh_token=eyJhbGciOiJIUzUxMiI...
```

Keycloak responds with a fresh access token, a fresh id token, **and a fresh refresh token** (rotation — covered in lecture 3). The old refresh token is invalidated.

### 4.8 Step 8 — Logout

```
GET http://localhost:8080/realms/crunch/protocol/openid-connect/logout
  ?id_token_hint=<the-id-token>
  &post_logout_redirect_uri=http%3A%2F%2Flocalhost%3A5173%2F
```

Keycloak terminates the session, clears its session cookie, and redirects back. The SPA clears its in-memory token state.

## 5. Decoding a JWT by hand — a guided exercise

Open the browser console on any page. Paste an ID token. Walk the segments.

```javascript
// 1. The raw token (truncate yours — this is just for the walk-through)
const token = "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJhbGljZSIsImlhdCI6MTczNDk5OTAwMH0.SIGNATURE";

// 2. Split into segments
const [headerB64, payloadB64, signatureB64] = token.split('.');

// 3. Helper: base64url decode
function b64urlDecode(s) {
  // Replace base64url chars with base64 chars, pad, decode
  const padded = (s + '===').slice(0, s.length + (4 - s.length % 4) % 4);
  return atob(padded.replace(/-/g, '+').replace(/_/g, '/'));
}

// 4. Decode header and payload
const header = JSON.parse(b64urlDecode(headerB64));
const payload = JSON.parse(b64urlDecode(payloadB64));

console.log('Header:', header);
console.log('Payload:', payload);

// 5. The signature is binary; printing it is not useful
console.log('Signature segment (base64url):', signatureB64);

// 6. Check expiration manually
const now = Math.floor(Date.now() / 1000);
console.log('Expired?', payload.exp < now);
```

You have now done what every JWT library does for you. Exercise 1 asks you to do exactly this against a token Keycloak issues to you.

## 6. Common verification mistakes

A short list of the failures that show up in production. Avoid each one.

1. **Trusting `alg` from the token's header without pinning.** Always specify an `algorithms` allowlist in your verifier.
2. **Skipping `aud` validation.** A token issued to a different client can be presented to yours; if you do not check `aud`, you accept it.
3. **Skipping `iss` validation.** A token from a different IdP can be presented to yours; if you do not check `iss`, you accept it.
4. **Allowing too much clock skew.** Five seconds is generous; sixty seconds is the upper end of reasonable. An hour is wrong.
5. **Not validating `nonce` on ID tokens.** Replay attacks become possible.
6. **Caching the JWKS for too long.** When the IdP rotates a key, your service rejects valid tokens until the cache expires. Cache for the discovery document's `cache-control` `max-age` value and **re-fetch on a kid miss**.
7. **Treating an expired token as a "soft" failure.** Reject hard, return 401, let the client refresh.
